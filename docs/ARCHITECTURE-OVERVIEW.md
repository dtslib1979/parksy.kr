## 3. 핵심 설계 원리

### 3-1. API 서버 없이 API처럼 동작

```
전통 방식:
  클라이언트 → API 서버 → 데이터베이스 → 응답
  (서버 비용 월 5만~50만원 + DB 비용 + SSL + 모니터링)

PARKSY 방식:
  클라이언트 → GitHub Pages → 정적 JSON 파일
  (비용: ₩0)
```

**어떻게 가능한가:**

GitHub Pages는 정적 파일을 웹으로 서빙한다.
JSON 파일을 `/api/v1/` 경로에 놓으면, 어디서든 `fetch()`로 가져올 수 있다.

```javascript
// 아무 웹페이지에서 이렇게 호출하면 끝
const data = await fetch('https://parksy.kr/api/v1/youtube.json');
```

서버 로직이 필요 없다. 데이터는 JSON 파일을 직접 수정한다.
수정하는 주체? **Claude Code**.

### 3-2. Claude Code = 만능 백엔드

전통적으로 백엔드 엔지니어가 하는 일:

| 작업 | 전통 | PARKSY |
|------|------|--------|
| API 엔드포인트 추가 | Express/Django + 배포 | JSON 파일 1개 생성 |
| 데이터 변환 | 서버 코드 작성 | Claude Code에게 시킴 |
| 버그 수정 | 개발자 → PR → 리뷰 → 배포 | Claude Code → git push |
| 새 페이지 생성 | 프레임워크 + 빌드 + 배포 | HTML 파일 1개 작성 |
| 배포 | CI/CD 파이프라인 | `git push` (GitHub Pages 자동 배포) |

**Claude Code가 API를 대체한다.**
서버가 요청을 받아서 처리하는 대신, Claude Code가 직접 파일을 생성/수정한다.
결과물은 GitHub에 커밋되고, GitHub Pages가 즉시 서빙한다.

### 3-3. 레포지토리 간 통신 = 반도체 버스라인

```
parksy-image/output/*.json  ──→  parksy.kr/api/v1/images.json
parksy-audio/output/*.json  ──→  parksy.kr/api/v1/audio.json
OrbitPrompt/prompts/*.json  ──→  parksy.kr/api/v1/prompts.json
```

각 레포지토리는 **반도체 공정의 한 단계**:

```
반도체 공정              PARKSY 공정
─────────               ───────────
웨이퍼 제조    =    Tistory 25개 (원자재 저장소, CDN)
전공정 (팹)    =    parksy-image, parksy-audio (소재 생산)
후공정 (패키징) =    parksy.kr (조립, 편성)
테스트         =    Claude Code (검증, QA)
출하           =    YouTube 5채널 (수익화)
```

레포 간 연결은 **JSON 파일이 버스라인** 역할.
별도의 메시지 큐(Kafka, RabbitMQ)나 API Gateway가 필요 없다.

## 4. 보안 분석

### 공격 불가능한 이유

| 공격 유형 | 전통 서버 | PARKSY |
|-----------|----------|--------|
| SQL Injection | DB가 있으면 위험 | **DB 없음. 불가능.** |
| 서버 해킹 | SSH/포트 노출 | **서버 없음. 불가능.** |
| API 키 탈취 | 서버에 키 저장 | **API 키 없음. 불가능.** |
| DDoS | 서버 다운 가능 | **GitHub CDN이 방어.** |
| XSS | 동적 렌더링 시 위험 | **정적 HTML. 최소 위험.** |
| 데이터 유출 | DB 덤프 가능 | **공개 JSON만 존재. 유출할 비밀 없음.** |
| 랜섬웨어 | 서버 파일 암호화 | **Git 히스토리. 언제든 복구.** |

### 유일한 보안 게이트

```
GitHub 계정 (2FA) → 레포지토리 접근 → 끝
```

하나만 지키면 된다. 공격 표면(attack surface)이 **1개**.

## 7. 3축 콘텐츠 교차 시스템

```
축 1: 한국 고교 교재 (dtslib1k)
  ─ 철학, 수학, 사회, 국어, 과학
  ─ 훅: "Korean education intensity"

축 2: KR Merit — 한국인 기질 (dtslib2k)
  ─ 허세교양, 하프블러드, 편집강박, 민주판타지
  ─ 훅: "Why Koreans are like that"

축 3: With-AI (전 채널 관통)
  ─ 프롬프트 공개 → AI 출력 → 페르소나 해설
  ─ 훅: "What happens when AI does this"
```

**소재 공식:**

```
페르소나(5) × 고교과목(5) × KR Merit(4) × AI = 100가지 조합

예시:
  Philosopher × 철학 × 민주판타지 × Claude
  = "AI로 한국 고교 철학 시험을 민주주의 판타지로 바꿔봤다"

  Musician × 국어 × korean-parksy × Lyria3
  = "한국 고전시가를 AI가 BGM으로 만들면"
```

100가지 조합 × 무한 에피소드 = **소재 고갈 불가능**

## 11. 장단점 정리

### 장점

1. **비용 0원** — 서버, DB, CDN, SSL, API 전부 무료
2. **보안 최강** — 서버 없으니 해킹 대상 없음. Git 히스토리로 언제든 복구
3. **확장 무제한** — JSON 파일 추가 = 새 API. HTML 파일 추가 = 새 페이지
4. **소재 무한** — 3축 교차 100+ 조합 × 무한 에피소드
5. **번아웃 없음** — AI가 양산, 사람은 편성/기획만
6. **수익 100% 본인** — MCN 수수료 없음
7. **데이터 완전 소유** — GitHub에 모든 코드/데이터/히스토리
8. **Claude Code = 무인력 백엔드** — API 서버 대체, 24시간 가용
9. **루프백 자동화** — 생산→배포→유입→재생산 사이클
10. **복제 불가** — 이 조합(AI + 한국 교육 + 한국 기질 + 방송국 구조)을 아는 사람 없음

### 단점 (솔직하게)

1. **GitHub Pages 트래픽 제한** — 월 100GB 대역폭. 대박 나면 마이그레이션 필요 (좋은 문제)
2. **실시간 기능 없음** — 채팅, 라이브 스트리밍은 YouTube에 의존
3. **검색 기능 없음** — 정적 사이트라 서버 사이드 검색 불가 (JS 클라이언트 검색은 가능)
4. **아직 가동 전** — 구조만 완성. 첫 영상 + 루프백 1회전이 진짜 검증
5. **1인 리스크** — 모든 의사결정이 한 사람. Claude Code가 보조하지만 최종 판단은 본인

### 단점이 단점이 아닌 이유

| 단점 | 왜 괜찮은가 |
|------|------------|
| 트래픽 제한 | 월 100GB면 수십만 방문 가능. 그 전에 수익 나서 마이그레이션 비용 충당 |
| 실시간 없음 | YouTube가 라이브 담당. 역할 분리 |
| 검색 없음 | 콘텐츠 25개 수준에서 검색 불필요. 성장 후 Algolia 무료 티어 |
| 미가동 | 이번 세션에서 시작 |
| 1인 리스크 | Claude Code가 사실상 2인 체제 |
